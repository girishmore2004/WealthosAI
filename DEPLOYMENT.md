# Deploying WealthOS AI

A practical, first-time-deploy guide: GitHub → Render (backend) → Vercel (frontend).
Everything here uses free tiers — no paid services required to get a working deployment.

> **Read this first**: an initial migration now exists at
> `packages/db/prisma/migrations/20260713000000_init/`, but it was **hand-derived from
> schema.prisma**, not generated by the real Prisma CLI — the environment this repo was
> built in has no network access to `binaries.prisma.sh` (Prisma's query-engine binary
> host), so `prisma migrate dev`/`diff` could never actually run there. A GitHub
> Actions job (`.github/workflows/ci.yml`, the `migration-check` job) applies this
> migration to a real Postgres instance on every push specifically to catch any
> mistake in that hand-derivation — check that it's green before trusting this in
> production. If you want full certainty before your first deploy, you can also
> regenerate it yourself in one command (real network access, real DB) — see Step 3.

---

## 1. Push the repo to GitHub

```bash
cd wealthos-ai
git init
git add .
git commit -m "Initial commit: WealthOS AI"
```

Create an empty repository on GitHub (no README/license — you already have files), then:

```bash
git remote add origin https://github.com/<your-username>/wealthos-ai.git
git branch -M main
git push -u origin main
```

Double-check `.gitignore` is doing its job before you push — you don't want `node_modules`,
`.env` files, or uploaded documents in the repo:

```bash
git status --short | grep -E "node_modules|\.env$|storage/documents"
# should print nothing
```

---

## 2. Provision a real Postgres + Redis (needed before Step 3)

You need a reachable `DATABASE_URL` to generate migrations and to run the app in
production. Free options:

- **Postgres**: [Neon](https://neon.tech) (free tier, serverless Postgres) or
  [Render's free Postgres](https://render.com/docs/free#free-postgresql).
- **Redis**: [Upstash](https://upstash.com) (free tier, serverless Redis) or Render's
  Redis add-on.

Either way, you'll end up with two connection strings that look like:

```
DATABASE_URL="postgresql://user:password@host:5432/dbname?sslmode=require"
REDIS_URL="rediss://default:password@host:6379"
```

Keep these handy — you'll paste them into both your local `.env` (for Step 3) and your
Render environment variables (Step 4).

---

## 3. Apply the migration (one-time, local)

```bash
cd wealthos-ai
npm install
cp apps/api/.env.example apps/api/.env
# edit apps/api/.env: paste your real DATABASE_URL and REDIS_URL from Step 2

cd packages/db
npx prisma migrate deploy
```

This applies the committed migration at `prisma/migrations/20260713000000_init/` to
your database. If it succeeds cleanly, you're done with this step.

**Want full certainty the migration is exactly right, not just hand-derived?**
Regenerate it for real — one command, now that you have real network access and a real
database:

```bash
rm -rf prisma/migrations
npx prisma migrate dev --name init
```

Prisma will either confirm the schema is already in sync (nothing to do) or generate a
fresh, engine-verified migration. Either way, commit whatever's in `prisma/migrations/`
afterward:

```bash
cd ../..
git add packages/db/prisma/migrations
git commit -m "Regenerate initial migration via real Prisma engine"
git push
```

Seed it now so you have something to look at once deployed:

```bash
npm run db:seed
```

(Alternative for a quick first release: `npx prisma db push` instead of
`migrate deploy` syncs the schema directly without touching migration history. Faster,
but you lose the ability to track/rollback schema changes over time — fine for a solo
demo, not recommended once other people depend on the data.)

## 4. Deploy the backend (NestJS) on Render

1. [render.com](https://render.com) → **New +** → **Web Service** → connect your GitHub repo.
2. Configure:

   | Setting | Value |
   |---|---|
   | **Root Directory** | *(leave blank — repo root, so npm workspaces can resolve `packages/db` and `packages/types`)* |
   | **Runtime** | Node |
   | **Build Command** | `npm install && npm run db:generate && npm run db:migrate:deploy && npm run build:api` |
   | **Start Command** | `node apps/api/dist/main.js` |
   | **Instance Type** | Free |

3. Add environment variables (Render dashboard → your service → **Environment**):

   | Key | Value |
   |---|---|
   | `DATABASE_URL` | your Postgres connection string from Step 2 |
   | `REDIS_URL` | your Redis connection string from Step 2 |
   | `SESSION_TTL_SECONDS` | `2592000` |
   | `API_PORT` | `4000` *(Render sets its own `PORT` too — see note below)* |
   | `WEB_URL` | your Vercel URL from Step 5, e.g. `https://wealthos-ai.vercel.app` *(you can leave this blank on first deploy and fill it in after Step 5, then redeploy)* |
   | `CORS_EXTRA_ORIGINS` | leave blank unless you have a second frontend origin (custom domain, staging site) — comma-separated |
   | `VERCEL_PREVIEW_PREFIX` | leave blank unless you want Vercel's per-branch/PR preview URLs to also work against this backend (see note below) |
   | `CROSS_SITE_COOKIES` | leave blank — defaults to `true` whenever `NODE_ENV=production`, which is correct for the standard Render + Vercel setup (see note below) |
   | `OTP_ADAPTER` | `mock` *(OTPs print to the Render logs)* or `resend` *(actually emails the code — see "Going further" below)* |
   | `RESEND_API_KEY` | required only if `OTP_ADAPTER=resend` — get one at [resend.com](https://resend.com) |
   | `OTP_FROM_EMAIL` | required only if `OTP_ADAPTER=resend` — must be a verified sender/domain in your Resend account |
   | `NODE_ENV` | `production` |
   | `DOCUMENT_STORAGE_PATH` | leave unset unless you've mounted a persistent disk (see note below) |

   > **Note on cross-site cookies**: Render (`*.onrender.com`) and Vercel (`*.vercel.app`,
   > or your custom domain) are two different domains, which makes every API call from
   > the frontend a **cross-site** request from the browser's point of view. Cross-site
   > cookies require `SameSite=None; Secure` — the more common `SameSite=Lax` setting is
   > silently dropped by the browser on cross-site `fetch`/XHR calls, which is what makes
   > login look like it "sometimes doesn't stick" (works on request, vanishes on reload).
   > `CROSS_SITE_COOKIES` defaults to `true` in production so you don't need to set it —
   > it's here as an override only for the less common case where frontend and backend
   > share a domain.

   > **Note on Vercel preview URLs**: by default, only the exact `WEB_URL` origin is
   > allowed through CORS — a PR/branch preview deployment (a different, randomly
   > suffixed `*.vercel.app` URL) will fail with a CORS error against your production
   > backend. Set `VERCEL_PREVIEW_PREFIX` to your project's preview-URL prefix (visible
   > in Vercel's deployment URL, e.g. `wealthos-ai-git-`) to allow those through too —
   > left blank, this matching is disabled entirely, which is the safer default.

   > **Note on `PORT`**: Render injects its own `PORT` env var and expects your app to
   > listen on it. `apps/api/src/main.ts` already checks `process.env.PORT` first (falling
   > back to `API_PORT` for local/self-hosted setups) — no changes needed here, just be
   > aware Render's `PORT` takes priority if you've also set `API_PORT`.

   > **Note on file storage**: Render's free tier filesystem is **ephemeral** — anything
   > written to `apps/api/storage/documents/` disappears on every redeploy/restart.
   > Document uploads will work between restarts but won't survive a redeploy. For a
   > real production deploy, either add a Render persistent disk (paid) or swap in an
   > S3-compatible adapter behind the existing `DocumentStorageAdapter` interface
   > (`apps/api/src/documents/adapters/`) — the interface was built for exactly this
   > swap.

4. Click **Create Web Service**. Watch the build logs — first build takes a few minutes
   (npm workspaces install + Prisma generate + Nest build).
5. Once live, note your backend URL: `https://<your-service>.onrender.com`.

## 5. Deploy the frontend (Next.js) on Vercel

1. [vercel.com](https://vercel.com) → **Add New** → **Project** → import the same GitHub repo.
2. Configure:

   | Setting | Value |
   |---|---|
   | **Root Directory** | `apps/web` |
   | **Framework Preset** | Next.js *(auto-detected)* |
   | **Build Command** | `cd ../.. && npm install && npm run build:packages && cd apps/web && npm run build` |
   | **Install Command** | *(leave default — the Build Command above handles install)* |
   | **Output Directory** | *(leave default — `.next`)* |

   The custom build command matters: `apps/web` depends on the workspace packages
   (`@wealthos/types`), so Vercel needs to build those first even though its root
   directory is `apps/web`.

3. Add environment variable:

   | Key | Value |
   |---|---|
   | `NEXT_PUBLIC_API_URL` | your Render backend URL from Step 4, e.g. `https://wealthos-api.onrender.com` |

4. Deploy. Once live, copy your Vercel URL and go back to Render (Step 4) to set
   `WEB_URL` to it, then trigger a redeploy on Render — this is what CORS
   (`apps/api/src/main.ts`'s `app.enableCors(...)`) uses to allow your frontend's origin.

### How the frontend finds the backend

`apps/web/lib/api-client.ts` reads `NEXT_PUBLIC_API_URL` at build time and prefixes
every API call with it. There's no other place a backend URL is hardcoded — changing
that one env var and redeploying the frontend is the only step needed to repoint it
(e.g. from a staging backend to production).

## 6. Post-deploy smoke test

1. Visit your Vercel URL → you should land on `/login`.
2. Enter any email → **Send login code** → if `OTP_ADAPTER=mock` (the default), check
   the Render service logs for a line like `[DEV ONLY] OTP for you@example.com: 482913`;
   if `OTP_ADAPTER=resend`, check the inbox for that email instead.
3. Enter the code → you should land on `/dashboard` with real (seeded, if you ran
   `db:seed`) data.
4. **Reload the page** (not just navigate within the app) — you should stay logged in.
   If you're bounced back to `/login` after a reload, the session cookie isn't
   surviving the cross-site request; double check `CROSS_SITE_COOKIES` isn't set to
   `"false"` and that both Render and Vercel URLs are `https://`.
5. Check the browser console/network tab for any CORS or 404 errors — if you see CORS
   errors, double check `WEB_URL` on Render matches your Vercel URL exactly (including
   `https://`, no trailing slash).

## Release checklist

Run this before every deploy, not just the first one. `.github/workflows/ci.yml` runs
steps 1–3 (plus a real migration-apply check) automatically on every push/PR to
`main` — this local checklist is for anyone who wants to verify before even pushing:

```bash
# 1. Consistency audit — schema, routes, braces, JSON
npm run audit

# 2. Full backend test suite
npm run --workspace=apps/api test

# 2b. Frontend test suite
npm run --workspace=apps/web test

# 3. Confirm the build actually compiles (catches type errors CI would miss otherwise)
npm run build:api
npm run build:web

# 4. If the schema changed: generate a migration locally and commit it
cd packages/db && npx prisma migrate dev --name <describe-the-change> && cd ../..
git add packages/db/prisma/migrations

# 5. Push — CI runs automatically; Render and Vercel both auto-deploy on push to main
# (if you left that on)
git push
```

On Render specifically, since the build command already runs `db:migrate:deploy`, a
schema change deploys itself — just make sure the migration file from step 4 above is
committed before you push, or Render will deploy the old schema.

## Going further (not required for a first deploy)

- **Real OTP delivery**: already wired — set `OTP_ADAPTER=resend` plus `RESEND_API_KEY`
  and `OTP_FROM_EMAIL` (see the Render env var table above). No code change needed. To
  add a different provider (Twilio, MSG91, ...), implement `OtpDeliveryAdapter`
  (`apps/api/src/auth/adapters/otp-delivery.adapter.ts`) and add a case for it in
  `otp-adapter.factory.ts`.
- **Real document storage**: swap `LocalDiskStorageAdapter`
  (`apps/api/src/documents/adapters/`) for an S3-compatible adapter behind the same
  `DocumentStorageAdapter` interface.
- **Real OCR**: swap `MockOcrAdapter` (`apps/api/src/documents/adapters/`) behind the
  same `OcrAdapter` interface.
- **Custom domain**: both Render and Vercel support this on their free tiers under
  their respective dashboards' Domain settings.
